How Madrona Makes One GPU Run Thousands of Simulated Worlds at Once

Teaching a simulated agent to play hide-and-seek — well enough to build forts and use tools — took about 115 billion separate tries. At a typical CPU simulator's speed of 1.6×10^5 steps per second, generating those tries takes about 40 days of nonstop computation. Every one of those tries has to come from a simulator, a program that computes what happens next when an agent acts. If that simulator is slow, the world's fastest neural-network hardware just sits idle waiting for data. This explainer shows how one research team rebuilt the simulator itself, not the learning algorithm, to run a million simulated steps a second on a single GPU. What does a training loop actually repeat? And what does a graphics chip really do when it claims to run things 'in parallel'?

The basic building blocks

Before we can see why a simulator becomes the bottleneck, we need two pieces of vocabulary: what an RL training loop repeats, and what a GPU can and cannot do fast. This part ends by watching the obvious 'just put it on the GPU' idea fail — which is exactly the problem the paper solves.

Reinforcement learning is a loop: act, observe, update, repeat

An agent acts, then the environment returns a new state and a reward. Repeated forever, that loop is reinforcement learning — watch the agent's reward climb, step by step.

GPU threads run in fixed groups of 32 and must all take the same branch

A GPU doesn't run 32 threads independently — it runs them in lockstep as one warp. Drag the divergence slider and watch the warp's effective speed collapse as threads are forced to take turns instead of moving together.

One environment per thread makes every warp diverge

It seems natural to give each simulated environment its own GPU thread. But real environments don't stay in sync — one agent bumps a wall while another glides freely. Turn up diversity and watch warp utilization crater, for the same reason as the last scene.

How Madrona actually works

With the ground rules in hand — training needs enormous step counts, and GPUs punish incoherent execution — we build up, piece by piece, the architecture that keeps thousands of simulated environments coherent on one chip.

Running all environments inside one program beats running many separate copies

Don't launch N separate simulators and hope the scheduler cooperates. Instead, run all N environments inside one program that sees them all at once — watch memory go from scattered to unified.

Entity-Component-System splits state into tables and logic into automatic per-row functions

Entities like Agent and Obstacle hold components — position, velocity, team. Entities with matching components collect into archetype tables. A system is one plain function that runs automatically on every matching row. Spawn entities and watch the tables and the system's sweep grow, with no new loop code written.

Packing each field into its own array lines up memory for parallel reads

Bundling one entity's fields together scatters memory when a system reads just one field across many entities. Packing each field into its own array lines those reads up — flip the toggle and watch addresses snap from scattered to contiguous.

One shared table across all environments beats one table per environment

Put every environment's same-type entities into one shared table, tagged by environment ID, instead of giving each environment its own. The same read becomes one contiguous sweep — toggle it and watch the jumps disappear.

One persistent GPU kernel beats hundreds of small kernel launches

Launching a separate GPU kernel — one program launched onto the GPU — for each of hide-and-seek's ≈200 systems means ≈200 doses of launch overhead and CPU synchronization every step. Compile them all into one persistent kernel instead, and warps grab the next task the instant they finish. Slide the system count up and watch the gap between the two timelines grow.

Splitting the kernel by register needs recovers lost GPU occupancy

One shared register budget — the register allocation given to every thread — means a heavy system (narrow-phase collision, up to 255 registers per thread) starves lighter ones of occupancy, the fraction of the GPU's thread slots actually filled. Profiling and splitting into separate sub-kernels fixes it — toggle it on and watch occupancy and throughput jump.

Parallelism inside one environment matters as much as parallelism across environments

Cross-environment parallelism isn't the whole story. Cartpole needs hundreds of thousands of environments to fill the GPU. Hide-and-seek's richer per-environment work — about 5 agents and 30 obstacles — fills the GPU at just a few thousand. Slide the batch size to compare the curves.

Why it's a breakthrough vs existing tech

Every design choice in part 2 was justified by a number. Now we look at the real workload — hide-and-seek with physics and ray-traced vision — and see exactly what those choices are worth against the naive GPU approach and against strong CPU engines.

Just using a GPU isn't enough — batching is what makes it fast

Just moving the naive one-thread-per-environment design onto a GPU doesn't help. On real hide-and-seek it hits only 1.9% of peak compute and bandwidth — slower than a CPU baseline. The fully batched design hits 33%/69% and is 42× faster. Select each architecture and compare the bars.

Madrona hits 1.9 million simulated steps per second on one GPU

Put it all together on the real workload: 1.9 million environment steps per second on one GPU. Select a task and compare against a strong 32-thread CPU baseline and the original open-source simulator.

Why now, and what it unlocks

The payoff isn't just a faster benchmark: it changes what experiments fit inside a single afternoon, and the same engine core already runs many different simulated worlds, not just one.

Faster simulation turns 40 days of training into a day and a half

Simulator speed converts directly into research time. At 1.6×105 steps/sec, the 115-billion-step run needed for hide-and-seek's full behavior takes about 40 days. At Madrona's 1.9×106 steps/sec, the same run takes about 1.5 days. Drag the throughput slider along the same curve.

One engine architecture already runs many different simulated worlds

The engine itself never changes — only the entities, components, and systems plugged into it do. The same batched-ECS core already runs hide-and-seek, Overcooked, Hanabi, and robotics tasks, each hitting peak throughput at a different batch size. Pick a task and watch the core stay fixed while the workload swaps in.
