ParallelKittens: closing the gap between how fast GPUs compute and how fast they can talk to each other

Training and running large AI models increasingly means splitting the work across many GPUs — but the wires connecting those chips have not kept pace with how fast the chips themselves compute. In real training runs today, GPUs can sit idle, waiting on each other, for over half the wall-clock time. This explainer walks through why that gap opened up, and how a small set of reusable code patterns closes it — turning a task that used to take thousands of lines of hand-tuned code into fewer than 50 lines, while matching or beating the best existing systems. To understand it honestly, we have to begin at the beginning: what a GPU actually looks like on the inside.

Before we can talk about splitting work across GPUs, we need the vocabulary for what's inside one — its threads, its memory tiers, its specialized circuits — and the historical trend that turned talking between GPUs into the slow part.

GPUs got much faster at computing than at talking to each other

Step through GPU generations: compute and local memory got 5-7x faster, but the links between GPUs only got 2-3x faster. That gap is why over half of some real training runs now goes to waiting.

How ParallelKittens actually works

With the hardware vocabulary in hand, we can follow the paper's three empirical findings — which transfer mechanism to use, how to split the chip's processors, and how much synchronization overhead is avoidable — and see how they're packaged into one small abstraction.

Total kernel time is the max of three costs, not their sum, once truly overlapped

Toggle from doing things one after another to doing them at once: total kernel time collapses from a sum of three costs to just their maximum, plus whatever leaks around the edges.

The best way to move data between GPUs depends on the message size

Slide the message size and watch which mechanism actually wins — the copy engine needs huge messages (≥256 MB) to hit its 82% peak, while TMA is already near peak at 2 KB.

Compute and communication can share one SM or split across separate SMs

Communication and compute can share threads inside one SM, or live on entirely separate SMs. Toggle the split and watch the synchronization cost jump from 64 ns on-chip to 832 ns through HBM.

The GPU network fabric can add numbers together as data passes through

The switch fabric between GPUs can sum data in transit instead of just moving it. Toggle to in-network reduction and watch the traffic — and the effective volume — collapse by roughly the GPU count.

Which scheduling strategy wins depends entirely on the workload

Pick the workload: for reduce-scatter, sharing an SM wins by 1.2x; for all-reduce, dedicating whole SMs wins by 3.6x. There's no single right answer.

The best split between compute and communication SMs shifts with problem size

Slide the matrix size and watch the best number of dedicated communication SMs shift — small matrices want more of them, huge ones want fewer.

Past a threshold reduction depth, communication is completely hidden behind compute

Compute time per tile grows with K; communication time per tile doesn't. Slide K past K≈2197 and watch unhidden communication collapse from 68% to under 1%.

Reading a peer GPU's memory caches the data on the peer, not on you

Reading a peer GPU's memory caches the data on the peer, not on you. Toggle from many redundant remote reads to one dedicated bulk prefetch and watch NVLink traffic drop to a single crossing.

One memory layout mirrored across all GPUs lets tile-based code reach across devices

One identically-shaped memory region, mirrored across every GPU and indexed by tile, is the handle for peer loads, stores, and reductions. Pick a collective and watch the added code stay under 50 lines.

Why it beats the status quo

Now that we know how it works, we can put it head-to-head against the libraries and hand-tuned kernels everyone currently uses, with the paper's own numbers.

Standard communication libraries add handshakes and buffer copies that most transfers don't need

Standard libraries add a round-trip handshake and an extra buffer copy on every transfer. Strip them for a direct one-way write and throughput rises 1.79x while access latency falls 4.5x.

Locking in one transfer mechanism can be slower than not overlapping at all

Lock a system into always using the copy engine, and small problems get slower than doing nothing at all. Slide the problem size and watch the hardwired curve dip below the no-overlap baseline.
