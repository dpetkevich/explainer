How to Run 16,000 Game Worlds at Once on One Graphics Card

Training an AI agent to play hide-and-seek took OpenAI 115 billion simulated practice steps. The expensive part wasn't the neural network — it was running the game itself. This paper stops simulating worlds one at a time and instead lays out every world's state in shared tables. That layout — not faster hardware — lets a single consumer GPU advance tens of thousands of worlds together at 1.9 million steps per second.

The bottleneck is the simulator

Before any clever engineering, we need the problem: AI agents learn by trial and error inside simulated worlds, and the simulator's speed sets the wall-clock cost of every experiment. We'll meet the hardware that could fix this — and see why the obvious way to use it fails.

Simulator speed, not learning math, sets the training clock

Hide-and-seek needed 115 billion simulated steps for tool use to emerge. Drag the simulator speed and watch months collapse into hours.

A GPU runs 32 threads in lockstep — divergence idles the hardware

An RTX 4090 keeps 30,000+ threads resident so that while some wait on memory, others swap in and keep the cores busy. But threads run in groups of 32 sharing one instruction stream — toggle the branch and watch half the lanes go dark.

Flip the loop: step every world's entities together

The GPU wants thousands of threads doing the same operation on adjacent data. Madrona delivers exactly that by borrowing a game-industry pattern — state as tables, logic as per-row functions — and merging every world into the same tables.

One table holds an archetype from every world — and warps love it

Each world has only 5 agents — never enough for a warp of 32. Slide the world count up and watch one shared Agent table feed every lane contiguous work.

The entire step is declared up front as a dependency graph

Hide-and-seek's step is ~200 system nodes wired by dependencies, known before the step begins. That foreknowledge is what lets the GPU run everything without asking the CPU for instructions.

One giant kernel: the GPU schedules all 200 nodes itself

A kernel is a program the CPU launches onto the GPU, and each launch costs fixed time. Launching 200 of them drowns tiny nodes in that overhead, and the collision step can't even launch until the CPU reads back how many collision pairs exist. One launch of self-scheduling warps removes both taxes.

Delete by marking, clean up by sorting

Every step, FindOverlaps spawns thousands of CollisionPair rows that die within that same step — a GPU memory allocator would choke. Watch deletion become a −1 tag and garbage collection become a sort.

The receipts

Same game logic, four ways of running it. The head-to-head numbers isolate which design choices actually buy the speed — and one final compiler trick squeezes out the last 35%.

Parallelism inside each world means you don't need millions of them

30,000 threads to feed, but each hide-and-seek world already contains hundreds of work items — collision pairs, 5 agents' observations, individual LIDAR rays. Slide the batch size and find the crossover yourself.

The last 35%: trading registers per thread for threads in flight

Registers come from a fixed pool per multiprocessor. More registers per thread means fewer threads resident to cover memory waits. One megakernel forces one budget on all 200 systems: the collision solver wants 255 registers; simple systems want 64 and 4× the threads. Move the budget and watch which systems win and lose.

What this unlocks

A 100× faster simulator doesn't just save money — it moves a supercomputer-scale experiment onto a desk, without asking the developer to write a single line of GPU code.

115 billion steps on the GPU under your desk — in ordinary sequential code

The old choice was flexible engines that step one world at a time, or hand-tuned GPU simulators frozen to one task. Flip the toggle: same experiment, 40 days of CPU time versus 1.5 days on a single RTX 4090.
