How to Simulate a Million Reinforcement-Learning Environments on One GPU

A reinforcement learning agent that discovers a genuinely new strategy — like building a fort out of movable boxes — needed on the order of 100 billion tries before that behavior showed up. If each try takes even a millisecond of simulated physics, that's years of wall-clock time no matter how good your learning algorithm is. This explainer builds up, from the ground rules of GPU hardware, the design that turns those hundred billion tries into a day and a half on a single consumer graphics card. To understand it honestly, we start with why simulators are slow, why GPUs are fussy about how they're used, and only then turn to the paper's actual machine.

The basic building blocks

Before judging any speedup we need the ground rules: how many tries reinforcement learning actually needs, how a GPU is willing to be used, and what it costs just to talk to one. These four facts explain why almost every existing simulator leaves nearly all of a GPU's power on the table.

Training complex behavior takes on the order of 100 billion tries

Complex behavior can take 1.15×1011 environment steps to emerge. Drag the simulator's speed and watch training time collapse from decades down to about a day.

Every GPU launch carries a fixed tax, however small the job

Every kernel launch costs the CPU a fixed microsecond-scale tax to launch and to wait for. Slide the step count up and watch that tax swallow the actual work.

Running many copies is not the same as running them together

Running N separate simulator copies gives the GPU nothing to coalesce; running all N inside one engine that knows about all of them turns identical operations into one coherent operation. Flip the switch and watch the access pattern turn from scattered red to coherent green.

The new technology

Given those constraints, the paper borrows a pattern from video-game engines — entity-component-system — and reshapes it entirely around the GPU's need for coherent memory and coherent instructions. Each scene below adds one piece of that machine.

Data and behavior are stored separately, and a query connects them

An entity — like one agent or one obstacle — is just a bundle of components (position, velocity, team). A system is one plain function that a query runs on every entity with the right components. Change the query and watch which rows light up.

One shared table for all environments beats one table per environment

Give every environment its own memory block and neighboring GPU threads read scattered addresses. Merge everyone's obstacles into one table tagged by environment ID and neighbors become adjacent. Flip the layout and watch the environment-count ceiling jump from about 8,000 to over 32,000.

Deleting an entity is a tag flip; reclaiming its memory is a sort

Deleting an entity just overwrites its environment tag with -1 — no locks, no shifting. Periodically a parallel sort shoves every -1 row to the end so the table can be truncated. Click delete, then run the sort, and watch the table shrink.

The whole simulation step runs as one self-scheduling kernel

Instead of ~200 separate kernel launches with a CPU round-trip between each, the whole graph is compiled into one kernel: warps just grab the next ready node from a shared pointer. Step through the graph and watch the launch-overhead slivers from before disappear.

Splitting the megakernel's register budget per system raises GPU utilization

One shared register budget for every system either starves cheap systems of parallelism or starves expensive systems of registers. Slide the budget for the selected system and watch resident-warp count trade off against speed — the paper's split raises compute use from 22% to 33%.

Why it beats existing simulators

With the machine built, we can put it head-to-head against both a naive GPU port and a strong CPU baseline, using the paper's own measured numbers.

One environment per GPU thread wastes almost the whole GPU

Put one whole environment on one GPU thread and you get exactly the failure from earlier scenes: only 1.9% of compute and memory bandwidth used, and a design 3× slower than a good CPU implementation. Toggle between throughput and utilization — Madrona wins both.

1.9 million steps per second turns 40 days of training into 1.5

Put it all together: 1.9 million steps per second on one RTX 4090. Pick a simulator and watch the same 1.15×1011-step training run take anywhere from months down to about a day and a half.

Why it matters now

The architecture's payoff isn't a fluke of one hand-tuned task — it generalizes across simple and complex environments alike, which is exactly the property reinforcement learning research needs as tasks keep getting more complex.

Entities inside one environment give the GPU work even at modest scale

A single environment already has dozens of entities to spread across threads, so Madrona doesn't need millions of environments to stay busy. Pick a task and watch how few environments it takes to hit 75% of peak throughput — 8,000 for a busy multi-agent task versus half a million for a nearly empty one.
