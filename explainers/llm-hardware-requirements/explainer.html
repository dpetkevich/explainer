The Arithmetic of Running a Giant AI Model

In July 2026, the lab Moonshot AI released Kimi K3, an AI model made of 2.8 trillion stored numbers. Ask a simple question — what machine can actually run this thing? — and you might expect the answer to require a PhD. It doesn't. Two lines of multiplication and division predict which models fit on a laptop, which need a data center, and why K3 had to be built the strange way it was. By the end, you'll be able to size the hardware for any AI model on the back of an envelope.

The arithmetic of running a model

Before we can size Kimi K3, we need to know what a model physically is and the two rules that govern it: one for how much memory it needs, one for how fast it can talk. We build both rules on a small model we can check by hand, then aim them at K3 — and hit a wall.

An AI model is billions of stored numbers, and nearly all of them are read for every word

A model is nothing but parameters — tuned numbers, 8 billion in this small model, 2.8 trillion in Kimi K3. Press play and watch: producing one token — a word-sized chunk of text — means sweeping through essentially all of them.

Each stored number costs a chosen number of bytes

Those 8 billion numbers each take up space, and engineers choose how much: from 4 bytes down to half a byte. Fewer bytes means each number is rounded to a coarser grid — like keeping fewer decimal places.

Rule 1: memory needed = number of parameters × bytes per parameter

Count times cost: that one multiplication is Rule 1. At 2 bytes per number, our small 8-billion-parameter model — it's called Llama 8B — needs 16 GB, its bigger sibling Llama 70B needs 140 GB, and at half a byte Kimi K3 still needs 1.4 TB.

Real machines come in memory tiers — and K3's 1.4 TB overflows every single box

Fitting is pass/fail; no processor speed rescues a model that doesn't fit. And K3's 1.4 TB overflows even the biggest H100 server — an 8-GPU node with 1.1 TB.

Rule 2: tokens per second ≈ memory bandwidth ÷ bytes read per token

Fitting is only half the story — remember from scene one that every token re-reads the whole model. Rule 2: divide the memory's reading speed by the gigabytes read per token, and you get the speed limit.

A conventional 2.8-trillion-parameter model would crawl at 2.4 tokens per second

A dense model — one that reads all its parameters for every token — at 2.8T moves 1.4 TB per token. An H100, a top data-center GPU, reads about 3.4 TB of memory per second, so that's roughly 2.4 tokens/s no matter how much memory you buy.

How Kimi K3 dodges the wall

Part 1 ended with a model that fits nowhere and, even if it did, would crawl at 2.4 tokens per second. K3's designers attack the arithmetic itself with three moves:
  • read only a sliver of the model per token (Mixture-of-Experts)
  • store each number in half a byte, trained that way from the start (4-bit training)
  • stop keeping a full record of the conversation (Kimi Delta Attention)

Mixture-of-Experts: store all 2.8 trillion numbers, but read only 2% per token

The dense wall came from reading everything per token, so a Mixture-of-Experts model routes each token to a few specialists instead. K3 picks 16 of its 896 experts — about 50B active parameters — so the read shrinks from 1.4 TB to 25 GB.

Squeezing a finished model to 4-bit throws information away

MoE fixed the reading bill; half-byte storage usually comes from quantization, rounding a finished model's numbers onto a coarser grid. That is not a lossless ZIP file — information is genuinely thrown away, and careless rounding makes the model dumber.

K3 is trained at 4-bit from the start, so the 4-bit file IS the model

The last scene's damage came from rounding a model that learned at full precision, so K3 trains on the 4-bit grid from the start. There is no fuller original to degrade; 1.4 TB is its native form.

The conversation itself eats memory: the KV cache grows with every token of context

Weights aren't the whole bill: while generating, the model keeps a record of every token of the conversation so far, called the KV cache. For Llama 70B, a full 128,000-token context costs about 40 GB — as much as its entire quantized weights.

Kimi Delta Attention keeps a running summary instead of a full per-token record

K3 advertises a 1-million-token context — about 8× the norm — so a record per token would make the KV cache the dominant cost. Kimi Delta Attention sidesteps that: it mostly keeps a fixed-size running summary of the conversation instead of a record per token.

The numbers, head to head

Each of K3's three moves changed one term in our two equations. Now we plug everything back in and compare: the impossible dense version of K3 against the real one, on real machines with real gigabytes.

Eight top-end GPUs technically hold K3's weights — and that's exactly why it needs 64

Eight B200s, the newest data-center GPUs, hold 1,536 GB — so the 1.4 TB of weights squeaks in with only ~136 GB to spare. But at K3's full 1-million-token context, the KV cache plus working memory comes to roughly 300 GB per conversation — more than double that leftover. That is why Moonshot recommends a 'supernode' of 64 GPUs: 64 × 192 GB ≈ 12 TB, room for the weights plus many conversations at once.

A 56× smaller read per token turns a 2.4 tokens/s crawl into fluid generation

Rule 2, head to head: the hypothetical dense 2.8T model reads 1.4 TB per token, and at an H100's 3.4 TB-per-second memory speed that is a 2.4 tokens/s crawl. The real K3 reads just 25 GB per token — 56× less — so the same 3.4 TB/s divided by 25 GB gives about 134 tokens per second on the supernode.

Why 2026, and what it unlocks

K3's design only became rational when new chips made 4-bit math the fast path rather than a storage trick — and the payoff shows up somewhere anyone can verify: the price list.

New chips compute in 4-bit natively, turning half-byte numbers from a storage trick into the fast path

Older GPUs could store 4-bit numbers but had to unpack them to 16-bit before doing math. Storage got cheaper; the math didn't — which is why nobody built K3 in 2023. Blackwell's B200 executes 4-bit math natively, making half-byte numbers the fastest path through the chip.

The arithmetic shows up on the price list: 50B-model prices for a 2.8T-parameter model

The arithmetic lands on a public price list: $3.00 per million input tokens and $15.00 per million output. That bill is sized by the 50B parameters K3 reads, not the 2.8T it stores. Cached input costs $0.30 — 10× cheaper — because a kept KV cache means repeated context never gets recomputed.
